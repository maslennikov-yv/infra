# Code Review (репозиторий infra)

Структурированное ревью с опорой на **C.L.E.A.R.**, правила `.cursor/rules` (в т. ч. **`general-engineering.mdc`**), **`README.md`** и README внутри каталогов сервисов. Итог — чеклист и список замечаний с приоритетами.

**Язык вывода:** чеклист и issues — на русском; технические термины, имена ресурсов Kubernetes и пути к файлам — как в репозитории.

---

## When to apply

Активировать skill, когда пользователь просит ревью кода, PR, ветки, модуля, сервиса в репозитории, аудит или проверку на соответствие правилам (см. также `.cursor/rules/code-review-triggers.mdc`).

---

## Доменные границы репозитория (для разбиения на модули)

Типичные «модули» для режимов A/B/C:

- Каталоги сервисов: **`postgres/`**, **`redis/`**, **`kafka/`**, **`minio/`**, **`clickhouse/`**, **`rabbitmq/`**, **`monitoring/`**, и т. д. — каждый со своим **`Makefile`**, **`values-$(ENV).yaml`**, локальным README.
- **Корень:** **`Makefile`**, **`helmfile.yaml.gotmpl`**, **`environments/`**, **`scripts/`**, **`apps/`** (**`registry.yaml`**, **`conf/_example/`**, скрипты merge/apply).
- Спец-модули: **`k8s-port-expose/`**, **`examples/`**, **`k8s/config/`** (локально; не секреты в git по `.gitignore`).

Суб-агентам передавать конкретные пути из этого списка, а не абстрактные «слои приложения».

---

## Scenarios

### A. Полный скан репозитория / большой фичи

1. Определить список модулей по каталогам сервисов и общим файлам (`helmfile`, `scripts`, `apps`).
2. Не распылять суб-агентов на десятки задач параллельно — группировать смежное (ориентир: несколько параллельных задач максимум).
3. По необходимости запустить суб-агентов с промптом из раздела «Prompt для суб-агента».
4. Собрать выводы: сквозные риски (дублирование values, несогласованность `ENV`, секреты, периметр), итоговый чеклист и единый список issues.

### B. Ревью PR / ветки

1. По диффу или списку файлов определить затронутые каталоги и корневые файлы.
2. Суб-агентов — только для затронутых областей с фокусом «изменения + границы с остальным репо».
3. Итог: чеклист + issues с учётом цели PR.

### C. Один модуль / сервис

- Небольшой объём — ревью самим агентом по чеклисту ниже.
- Крупный каталог — один суб-агент на этот путь.

## Sub-agent usage

- Режим чтения или полного доступа — по задаче.
- Главный агент агрегирует структурированный ответ (чеклист + issues), не подменяет его «общими советами».

### Prompt для суб-агента

```
Проведи код-ревью для области [имя] (пути: [конкретные каталоги/файлы, например postgres/, scripts/apps-apply.sh, helmfile.yaml.gotmpl]).

Используй C.L.E.A.R.: Context (цель изменений, как релизы и конфигурация влияют на кластер и окружения), Layered examination (структура чартов/Makefile → логика целей и шаблонов → безопасность и периметр → ресурсы/стоимость → стиль и поддерживаемость), Explicit verification для критичных мест (apply, секреты, PVC, ingress), Alternative consideration и Refactoring recommendations при необходимости.

Проверь соответствие правилам репозитория infra: общий Makefile и include environments/$(ENV).mk; Helm/helmfile согласованы; values явные (образы, ресурсы, persistence); секреты и apps/conf не утекают в git; деструктивные операции по докам сервиса; перед apply где уместно — diff/template; kubectl/Make сообщения с контекстом; KISS/YAGNI и отсутствие лишнего рефакторинга соседей.

Верни:
1) Заполненный чеклист по области (контекст, C.L.E.A.R., правила infra).
2) Замечания с приоритетами Critical / Suggestion / Nice to have — описание, путь, рекомендация.
```

## Checklist (C.L.E.A.R. + правила infra)

### Контекст

- [ ] Понятна цель ревью и границы диффа/модуля.
- [ ] Учтены целевое окружение (`ENV`), namespace/релизы и влияние на данные.

### C.L.E.A.R.

- [ ] **Уровень 1 — структура:** организация каталога сервиса, связь с корневым `Makefile` / `helmfile`, отсутствие «магии» вне принятых точек входа.
- [ ] **Уровень 2 — логика:** цели Make, параметры helmfile/values согласованы; скрипты в `scripts/` проверяют предусловия и не глотают ошибки без причины.
- [ ] **Уровень 3 — безопасность и граничные случаи:** секреты и `apps/conf` вне примера не в git; ingress и админ-UI осознанны; учётки приложений изолированы по namespace/Secret как в доке.
- [ ] **Уровень 4 — ресурсы и эксплуатация:** лимиты/запросы, PVC, последствия reset — явны; симптомы не маскируются костылями в values.
- [ ] **Уровень 5 — стиль:** DRY там, где принято в репо (общие цели Make, шаблоны helmfile); дублирование `values-*` между окружениями осознанно; без mass-sed по values.

### Правила проекта (сверка с `general-engineering.mdc`)

- [ ] **apps/:** живые конфиги не коммитим; реестр `apps/registry.yaml`; у `enabled: true` нет дублирующихся `name`.
- [ ] **Конфигурация:** `values-$(ENV).yaml`, `environments/$(ENV).mk`, явные теги/registry; перед критичным применением — `make diff` / эквивалент.
- [ ] **Инструменты:** изменения воспроизводимы через документированные команды Makefile/README.

## Output format

### 1. Чеклист

Заполненный чеклист выше с отметками и краткими комментариями.

### 2. Issues

Для каждого замечания: **Приоритет** (Critical | Suggestion | Nice to have), **Описание**, **Путь** (файл, при необходимости цель Make или ресурс), **Рекомендация**.

**Шаблон:**

```markdown
- **Critical** | [Краткое описание]
  - Путь: `postgres/Makefile` / `postgres/values-prod.yaml`
  - Рекомендация: [конкретное действие]
```

---

## Reference: C.L.E.A.R.

### 1. Context Establishment

Цель изменений, границы, встраивание в Kubernetes/helmfile/Make, требования по безопасности и доступности.

### 2. Layered Examination

Слои перечислены в чеклисте C.L.E.A.R. выше — применять к манифестам, values, Makefile, скриптам.

### 3. Explicit Verification

Для изменений с `apply`, удалением данных, секретами, публикацией сервисов — мысленно прогнать порядок операций и откат.

### 4. Alternative Consideration

Избыточность абстракций, альтернативы (например, отдельный chart vs переиспользование values), соответствие KISS репозитория.

### 5. Refactoring Recommendations

Конкретные шаги с путями; приоритет Critical → Suggestion → Nice to have.

## Типовые области (замена «HTTP/ORM» из прикладных репо)

### Helm / helmfile / values

- [ ] Семантика окружений согласована; нет скрытых дефолтов, ломающих prod.
- [ ] Имена релизов, namespace, зависимости чартов согласованы с документацией сервиса.

### Makefile (корень и сервис)

- [ ] `KUBECONFIG`, `ENV`, SSH-переменные используются так же, как в остальном репо.
- [ ] Цели документированы или следуют паттерну соседних сервисов.

### Скрипты (`scripts/`)

- [ ] Входные пути и переменные окружения явны; ошибки с контекстом для оператора.

### apps / периметр

- [ ] В дифф не попали секреты; изменения в `registry` и шаблонах конфигов согласованы с `scripts/apps-*.sh`.
