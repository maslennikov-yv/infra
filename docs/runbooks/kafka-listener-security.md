# Kafka listener security: текущее состояние и план миграции на SSL

## TL;DR

В `kafka/values-{local,prod}.yaml` сейчас:

```yaml
listeners:
  client:      { protocol: SASL_PLAINTEXT }   # SCRAM-auth, но трафик в plaintext
  controller:  { protocol: PLAINTEXT }        # KRaft quorum без auth и без шифрования
  interbroker: { protocol: PLAINTEXT }        # межбрoker трафик без auth и без шифрования
```

Сделано так осознанно из-за **bootstrap chicken-egg**: SCRAM-юзеры
хранятся внутри KRaft metadata, и controller'ы не могут поднять quorum
с `SASL_*`-протоколом до того, как первый user будет записан в metadata.
Комментарий в `values-prod.yaml:21-23` фиксирует это.

CNI кластера — Calico, NetworkPolicy поддерживается. Текущая первая
линия защиты — **NetworkPolicy** (см.
`docs/runbooks/network-policy.md`): если она будет настроена с
`allowExternal: false`, доступ к kafka 9092/9094 будут иметь только
помеченные namespace.

## Threat model

**Что может злоумышленник внутри кластера сейчас:**

- Pod в любом namespace → видит controller (`9094`), interbroker (`9094`)
  и client (`9092`) трафик через tcpdump на shared network.
- Метаданные KRaft (топики, ACL, конфиги, SCRAM-credentials hash) идут
  по controller listener в plaintext. Sniff даст полную картину.
- Подмена controller-сообщений (без auth) теоретически возможна, но
  требует MITM на уровне сети (Calico это маловероятно, но всё-таки).
- Client listener: SCRAM-handshake **не утечёт** пароль (challenge-
  response защищает от eavesdropping), но **сами client requests/
  responses** (produce, consume, describe-topics) видны.

**Какой риск принимать прямо сейчас:**

- Для prod без NetworkPolicy + кластер с приложениями разных команд →
  риск **высокий**, рекомендуется миграция.
- Для local/dev и prod с настроенной NetworkPolicy (`allowExternal:
  false`) → риск **умеренный**, миграция в бэклоге.

## Целевое состояние

```yaml
listeners:
  client:      { protocol: SASL_SSL }   # SCRAM-auth + TLS-шифрование
  controller:  { protocol: SSL }        # mTLS между controllers, без SASL
  interbroker: { protocol: SSL }        # mTLS между brokers
```

Почему **SSL** (не `SASL_SSL`) для controller — обходит chicken-egg:
mTLS использует X.509-сертификаты, которые лежат в Secret вне KRaft
metadata, и поднимают quorum независимо от SCRAM-юзеров.

Client listener остаётся `SASL_*`, чтобы не ломать существующих
клиентов: они продолжают аутентифицироваться по username/password из
Secret `<APP>-kafka` (создаётся `make kafka-app-create`), но трафик
теперь шифруется TLS.

## План миграции

Миграция через **полный recreate с restore из backup-meta**, потому что
KRaft не поддерживает rolling переход с PLAINTEXT-controller на
SSL-controller (controllerListenerNames одно, и не может быть двух
протоколов одновременно). Это **downtime ~5-15 минут** для prod.

Альтернатива «zero-downtime» (dual-listener через `extraListeners`)
требует параллельного запуска двух наборов controllers с разными
ролями и не поддерживается Bitnami chart out-of-the-box. Для большинства
сценариев простой recreate достаточен.

### Шаг 0. Maintenance window

Сообщить владельцам приложений, что Kafka будет недоступна ~15 минут.

### Шаг 1. Backup metadata и offsets

```bash
make kafka-backup-meta ENV=prod
# Производит kafka/backups/kafka-meta-YYYYMMDD-HHMMSS.tar.gz:
# топики (с конфигом), consumer groups, offsets, ACL.
```

### Шаг 2. Подготовить TLS

Bitnami chart умеет генерировать сертификаты сам. Два варианта:

**Вариант A — `tls.autoGenerated.engine: helm`** (просто, но certs
обновляются только при `helm upgrade`):

```yaml
# kafka/values-prod.yaml — добавить
tls:
  type: PEM
  autoGenerated:
    enabled: true
    engine: helm
```

**Вариант B — `cert-manager`** (рекомендуется для prod, ротация автоматом):

```yaml
tls:
  type: PEM
  autoGenerated:
    enabled: true
    engine: cert-manager
    certManager:
      existingIssuer: "letsencrypt"   # или ваш ClusterIssuer/Issuer
      existingIssuerKind: ClusterIssuer
      duration: 2160h    # 90 дней
      renewBefore: 360h  # обновлять за 15 дней до истечения
```

Если cert-manager в кластере не установлен — установите или используйте
Вариант A.

### Шаг 3. Recreate с новым конфигом

```bash
make kafka-recreate-prep ENV=prod
# Сохраняет Secret kafka-kraft, удаляет StatefulSet и PVC.
# (Дальнейший helmfile apply пересоздаст с новым SSL-конфигом.)
```

В `kafka/values-prod.yaml` перед `helmfile apply` поменять:

```yaml
listeners:
  client:      { protocol: SASL_SSL }
  controller:  { protocol: SSL }
  interbroker: { protocol: SSL }

# Для controller mTLS (controllerListenerName требует Authenticated principal)
overrideConfiguration:
  authorizer.class.name: org.apache.kafka.metadata.authorizer.StandardAuthorizer
  super.users: User:user1;User:CN=kafka-controller    # CN из автогенерированного cert
  allow.everyone.if.no.acl.found: "false"
  ssl.client.auth: required                            # mTLS — сервер требует сертификат от клиента
  ssl.endpoint.identification.algorithm: ""            # отключить hostname-verification внутри cluster
```

`super.users: User:CN=kafka-controller` — точное значение зависит от
того, какой CN пишет Bitnami в auto-generated certs (обычно `CN=
<release>-controller`). Проверить через `openssl x509 -in
<cert> -text -noout | grep Subject` после rendering.

### Шаг 4. Apply и восстановление

```bash
make up ENV=prod ENABLED_SERVICES=kafka
# helmfile apply создаст новый StatefulSet с TLS, controllers поднимут
# quorum по mTLS, broker register'ятся с SCRAM-credentials, которые
# чарт перепишет из existingSecret kafka-user-passwords.

make kafka-restore-meta-topics ENV=prod
# Восстанавливает топики, configs, ACL из backup-meta.
# Offsets и consumer-groups можно пропустить (consumers сами догонят
# по auto.offset.reset) или восстановить отдельно.

make kafka-status ENV=prod
# Проверка: 3 controller-pod в READY, NodeID-ы записаны в metadata.
```

### Шаг 5. Smoke-тест

```bash
# Убедиться, что client listener теперь требует TLS:
kubectl run -it --rm -n default kafka-test --image=alpine --restart=Never -- \
  sh -c 'apk add -q openssl && echo | openssl s_client -connect kafka.kafka:9092 2>&1 | head -20'
# Должен показать handshake (TLS 1.2/1.3, server cert).

# Topic list через SASL_SSL (потребует client.properties с TLS truststore):
make kafka-topic-list ENV=prod
# Команда внутри pod'а — использует существующий Secret с client cert.
```

### Шаг 6. Apps-перенастройка

Каждое приложение, подключающееся к Kafka, должно:

- Использовать `security.protocol=SASL_SSL` вместо `SASL_PLAINTEXT`.
- Указать `ssl.truststore.location` (или `ssl.ca.location` для librdkafka)
  с CA-сертификатом — он лежит в том же Secret, что keystore.
- Не менять SCRAM username/password (они те же).

`make kafka-app-create APP=… ENV=prod` после миграции должен будет
включать в Secret и CA-cert. Это требует обновления
`scripts/app-create-in-pod.sh` или прямого `kubectl create secret`
после apply. **Эта правка — отдельный TODO**, runbook её не покрывает.

## Smoke-тест

После apply смотрим:

- `make kafka-status ENV=prod` — все 3 controller READY.
- `kubectl logs -n kafka kafka-controller-0` — нет ошибок типа `SSL
  handshake failed`, `Authentication failed for ...`.
- `kubectl exec -n kafka kafka-controller-0 -- kafka-metadata-shell.sh
  --snapshot /bitnami/kafka/data/__cluster_metadata-0/00000000000000000000.log
  --print` — список топиков должен совпадать с `kafka-backup-meta`.

## Rollback

Если что-то пошло не так:

1. **Быстрый**: вернуть в `kafka/values-prod.yaml` старые protocols
   (`PLAINTEXT` для controller/interbroker, `SASL_PLAINTEXT` для client),
   убрать `tls:` блок, `make kafka-recreate-prep` + `make up
   ENABLED_SERVICES=kafka` + `make kafka-restore-meta-topics`.
2. **Сертификаты сохранить** — если recreated cert-manager-managed,
   они автоматически создадутся заново; если helm-engine — потеряются
   и сгенерятся заново при следующем apply.

## Riski

- **Downtime**: ~5–15 минут. Все consumers/producers получают
  `OFFLINE_BROKERS` пока StatefulSet не подымется заново.
- **Apps требуют обновления конфига** для TLS (см. Шаг 6) — без этого
  они не подключатся.
- **`super.users`-CN зависит от engine**: нужно проверить точный
  Subject CN в auto-generated cert до apply, иначе controllers не
  будут признавать друг друга и quorum не поднимется.
- **Нет dual-listener migration в Bitnami chart** — это recreate
  процедура, не rolling. Если zero-downtime обязателен, рассмотрите
  Strimzi Operator (отдельный проект, не совместим с этим чартом).

## Следующие шаги (бэклог после миграции)

- **SASL_SSL для client** — уже включается этим runbook'ом.
- **Авто-ротация client SCRAM-passwords** — отдельная процедура,
  использует `kafka-configs.sh --alter --add-config 'SCRAM-SHA-256=...'`.
- **Audit log** — `log4j.properties` с уровнем INFO для
  `kafka.authorizer.logger`, чтобы видеть отказы ACL.
