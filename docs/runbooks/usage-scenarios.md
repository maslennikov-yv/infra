# Сценарии использования infra: local/prod и приложения

Описательные потоки для оператора кластера (microk8s), который поднимает data-сервисы и выдаёт приложениям изолированные учётки. Длинные списки команд не дублируются — есть ссылки на [README](../../README.md) и смежные документы. Сценарии расположены по типичной последовательности применения: бутстрап → ежедневная работа с приложениями → регулярное обслуживание → редкие операции.

---

## infra-lab: общие замечания

Запуск: `npm ci && node scripts/infra-lab.mjs`. Двухуровневая навигация: **задача** (Бутстрап / Конфигурирование / Управление) × **объект** (Сессия / Среда / Сервис / Приложение). Helm-`values` и сами чарты правятся в репозитории, не через TUI.

- **ENV сессии:** **Конфигурирование → Сессия** ([session-env-picker.mjs](../../scripts/lib/session-env-picker.mjs)) — выбранное окружение действует для всего сеанса.
- **Глобальный Helm `up` / `diff` / `down`:** после действия — мультивыбор `ENABLED_SERVICES` (или «весь набор»), затем `EXCLUDE_SERVICES`. Для `up` дополнительно: пропуск `apps-apply` (`SKIP_APPS_APPLY=1`), продолжать при ошибках (`APPS_APPLY_CONTINUE_ON_ERROR=1`).
- **`apps-apply`:** **Управление → Приложение → Применить учётки** — те же фильтры `ENABLED_SERVICES` / `EXCLUDE_SERVICES`.
- **Закрепление состава за ENV:** после глобального Helm или `apps-apply` TUI предлагает сохранить выбор в `environments/<ENV>.mk` ([env-mk-persist.mjs](../../scripts/lib/env-mk-persist.mjs)); либо правка вручную.

Подписи пунктов и ветвления совпадают с TUI — расхождения с этим документом считают багом документации или меню.

## Сценарий 1: бутстрап нового окружения с нуля

**Цель:** развернуть инфру на новой ноде (свежий microk8s) либо ввести новое `ENV` (например `staging`).

1. Рыба окружения — `make env-new ENV=…` (`environments/<ENV>.{mk,yaml}`, `k8s/config/<ENV>`, `values-<ENV>.yaml` в каждом сервисе). TUI: **Бутстрап → Среда → «Создать рыбу окружения»**.
2. Доступ к кластеру — TUI: **Бутстрап → Среда → Кластер и доступ**:
   - **Удалённо:** `SSH_HOST` / `SSH_KEY` в `environments/<ENV>.mk`, затем `microk8s-setup` + `kubeconfig-fetch`.
   - **Локально:** `kubeconfig-microk8s-local` (без SSH).
3. Образы: `make images-save ENV=…` → `make images-push-remote ENV=…` (удалённо) или `make images-push ENV=…` (local). TUI: **Управление → Среда → Образы контейнеров** (с опц. фильтром по `SERVICE`).
4. Развёртывание: `make up ENV=…` (вместе с `apps-apply`, если не `SKIP_APPS_APPLY=1`). TUI: **Управление → Среда → Релизы Helm: весь набор** (см. [общие замечания](#infra-lab-общие-замечания)).
5. Sanity: `make doctor ENV=…`. TUI: **Управление → Среда → Кластер и доступ → «Полная диагностика стека»**.

Восстановление существующего ENV из git + бэкапов — [disaster-recovery.md](./disaster-recovery.md).

## Сценарий 2: новое приложение с подмножеством сервисов

**Цель:** приложению нужны, например, только Postgres и Redis — без Kafka/ClickHouse.

1. Платформенные сервисы уже подняты (см. сценарий 1).
2. Шаблон каталога `apps/conf/<app>/` + опционально запись в `apps/registry.yaml` — `make apps-conf-template APP=<app>`. TUI: **Бутстрап → Приложение → «Создать каталог из шаблона»** или **Конфигурирование/Управление → Приложение → Конфигуратор** (режим «Новое приложение»).
3. Заполнить секреты и поля (`enabled`, `app_ns`, `redis_db`, `repo`) — конфигуратор. Просмотр merge: `make apps-merge-print` (TUI: **Конфигурирование → Приложение → «Показать итоговую конфигурацию»**).
4. Применить в кластер — `make apps-apply ENV=…`. TUI: **Управление → Приложение → «Применить учётки и конфиги»**.
5. Сервис-специфичные операции — цели `*-app-create` / `*-app-show-creds` / `pg-app-psql` и пр. TUI: **Управление → Приложение → «Логины приложений в БД и брокерах»** → движок.

Для `ENV=local` с hostPath исходников: `make apps-local-src-helm-sets ENV=local APP=…` (TUI: **Конфигурирование → Приложение → «Вывести helm --set для local hostPath»**) — строки `--set` для чарта приложения; шаблон тома задан в самом чарте.

## Сценарий 3: смена учёток, ротация секретов, правка квот

**Цель:** новый пароль приложения, другой Redis DB, другие параметры топика Kafka.

1. Новые секреты в `apps/conf/<app>/` через конфигуратор; применение — `make apps-apply ENV=…` (TUI: **Управление → Приложение → Применить…**).
2. Просмотр текущих кредов: `*-app-show-creds`. TUI: **Управление → Приложение → Логины…** → движок → «Показать…».
3. Kafka-топики: `kafka-topic-alter` / `-describe` / `-list`. TUI: **Управление → Приложение → «Топики Kafka»** (для топиков по APP) либо **Управление → Сервис → Kafka** (общая админка топиков и сброс данных).

При ротации секретов и правках `apps/registry.yaml` — обновить env-архив (см. сценарий 5).

## Сценарий 4: здоровье сервисов и инциденты

**Цель:** понять статус релиза и упирается ли расчёт в ресурсы ноды.

1. Комплексная сводка: `make doctor ENV=…` (тулинг + кластер + helm vs helmfile + rollouts + per-app verify). TUI: **Управление → Среда → Кластер и доступ → «Полная диагностика»**.
2. Per-сервис: `<svc>-status` / `-logs` / `-shell`. TUI: **Управление → Сервис → Диагностика сервисов** → компонент.
3. Нагрузка ноды: `top-totals`, `monitoring-top-nodes`, `monitoring-pod-events`, `monitoring-describe-pod`. TUI: **Управление → Среда → Кластер и доступ** или **Диагностика → Netdata → «Ещё»**.
4. Постоянный слой наблюдаемости — [Netdata](../../README.md#мониторинг-netdata) в `ENABLED_SERVICES` (включается мультивыбором глобального Helm).

## Сценарий 5: регулярные бэкапы и проверка восстановления

**Цель:** свежие бэкапы стейтфул-сервисов и периодический smoke-тест применимости.

1. По графику (cron / systemd timer / CI) для каждого `ENV`:
   - **Все стейтфул-сервисы:** `make backup-all ENV=…` — последовательно `postgres-backup`, `redis-backup`, `kafka-backup-meta`, `minio-backup-meta`, `clickhouse-backup`, `rabbitmq-backup-defs` (учитывает `ENABLED_SERVICES`/`EXCLUDE_SERVICES`). Артефакты — в `<service>/backups/` (gitignored). Подробности — `<service>/BACKUP.md`. TUI: **Управление → Среда → «Бэкап всех stateful-сервисов разом»**.
   - **Платформенные Secrets и `apps/conf/`:** `make env-backup ENV=…` — `environments/backups/<env>-YYYYMMDD-HHMMSS.tar.gz`. Без него восстановление с нуля невозможно. TUI: **Управление → Среда → «Сохранить … в архив»**.
2. Хранить артефакты **вне той же ноды** (rsync/rclone в S3, отдельный сервер, при необходимости шифрование — см. [secrets-management.md](./secrets-management.md), [onboarding-admin.md](../onboarding-admin.md)).
3. Ротация — оператор задаёт RPO/RTO, чистит старые архивы.
4. **Smoke-тест восстановления** в local/staging: `make <svc>-restore BACKUP_FILE=… ENV=…` (для каждого сервиса — соответствующий `<service>/BACKUP.md`). TUI: **Управление → Сервис → Диагностика → `<svc>` → «Бэкапы и восстановление»** — запрос файла, явное подтверждение. Без smoke-теста «бэкап есть» ≠ «бэкап применим».
5. При смене паролей/ACL и правках `apps/registry.yaml`/`apps/conf/<app>/` — новый `env-backup`, иначе старый архив содержит устаревшие Secrets.

Расписание задаётся в системе оператора, не в TUI. Полный DR-runbook — [disaster-recovery.md](./disaster-recovery.md).

## Сценарий 6: добавить сервис в работающий кластер

**Цель:** раньше были только Postgres/Redis; теперь нужен Kafka.

1. Расширить состав в `environments/<ENV>.mk` (`ENABLED_SERVICES` / `EXCLUDE_SERVICES`).
2. Для Kafka: `make kafka-bootstrap`. TUI: **Бутстрап → Сервис → «Инициализировать подключение к Kafka»**.
3. Образы для нового компонента: `make images-save/push ENV=… SERVICE=<svc>`. TUI: **Управление → Среда → Образы контейнеров** → «только один data-сервис?» → да.
4. Развёртывание: `make diff ENV=…` → `make up ENV=…` (или точечно `<svc>-up`). TUI: **Управление → Сервис → Helm → Один компонент**.
5. Учётки приложений: `<svc>-app-create`. TUI: **Управление → Приложение → Логины…** → движок.

## Сценарий 7: две среды с разным составом сервисов

**Цель:** на local — лёгкий стек (например, без Kafka и ClickHouse), на prod — полный (или наоборот: на local всё под интеграционные тесты).

1. Зафиксировать состав в `environments/<ENV>.mk` (`ENABLED_SERVICES` / `EXCLUDE_SERVICES`) — см. [README — окружения](../../README.md#окружения-localprodstaging).
2. Образы и Helm — как в сценарии 1, с учётом ENV сессии (TUI: **Конфигурирование → Сессия**).
3. После каждой массовой операции TUI предлагает сохранить набор сервисов в `<ENV>.mk` — соглашайтесь, чтобы политика была устойчивой.

**Итог:** одна кодовая база, разные `values-<ENV>.yaml` и разный набор релизов.

## Сценарий 8: убрать или отключить сервис или приложение

**Цель:** разгрузить local или вывести приложение из эксплуатации.

- **Приложение:** в реестре `enabled: false`, затем `make apps-apply`; при необходимости явное удаление учёток (`*-app-drop` с подтверждением). TUI: мастер **«Деактивировать приложение в registry и снять учётки»** (Конфигурирование/Управление → Приложение).
- **Платформенный сервис:** сузить `ENABLED_SERVICES` или расширить `EXCLUDE_SERVICES`, `make diff` / `make up`; либо точечно `<svc>-down`. TUI: **Управление → Сервис → Helm → Один компонент → «Уничтожить релиз»**. Осторожно с PVC — предупреждения в `<service>/README.md`.
- **Полный снос окружения:** `make down ENV=…`. TUI: **Управление → Среда → Релизы Helm: весь набор → Уничтожить**.

Отделяйте «снять с helm» от «очистить учётки и данные приложения» — это разные шаги.
