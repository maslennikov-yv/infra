# Сценарии использования infra: local/prod и приложения

Описательные потоки для оператора кластера (microk8s), который поднимает data-сервисы и выдаёт приложениям изолированные учётки. Длинные списки команд не дублируются — есть ссылки на [README](../../README.md) и смежные документы. Сценарии расположены по типичной последовательности применения: бутстрап → ежедневная работа с приложениями → регулярное обслуживание → редкие операции.

---

## infra TUI: общие замечания

Запуск: `npm ci && make infra` (или `node scripts/infra.mjs`). Action-first меню из 13 пунктов в группах: **Каждый день** / **Сценарии** / **Бэкапы** / **Настройка** / **Сессия**. Helm-`values` и сами чарты правятся в репозитории, не через TUI.

- **Контекст сессии:** в шапке меню видны `ENV` и `APP`. `ENV` выбирается через **Сессия → Сменить ENV** (или при первом запуске берётся из `defaultEnv()`); `APP` — через **Сессия → Сменить APP**, кэшируется per-ENV в `~/.cache/infra-tui/last-app-<env>` и подставляется во все действия, где он нужен.
- **Глобальный Helm:** **Каждый день → Применить изменения / Сравнить с кластером**. Мультивыбор `ENABLED_SERVICES` (или «весь набор») + опц. `EXCLUDE_SERVICES`. Для `up` дополнительно: пропуск `apps-apply` (`SKIP_APPS_APPLY=1`), продолжать при ошибках (`APPS_APPLY_CONTINUE_ON_ERROR=1`).
- **`apps-apply`:** **Каждый день → Учётки и топики → Применить учётки** — те же фильтры `ENABLED_SERVICES` / `EXCLUDE_SERVICES`.
- **Закрепление состава за ENV:** после **Применить изменения** TUI предлагает сохранить выбор в `environments/<ENV>.mk` ([env-mk-persist.mjs](../../scripts/lib/env-mk-persist.mjs)); полноценное управление политикой — **Сценарии → Конфигурирование сервисов** (см. Сценарий 6).

Подписи пунктов и ветвления совпадают с TUI — расхождения с этим документом считают багом документации или меню.

## Сценарий 1: бутстрап нового окружения с нуля

**Цель:** развернуть инфру на новой ноде (свежий microk8s) либо ввести новое `ENV` (например `staging`).

Отдельного wizard'а нет — это последовательность пунктов TUI, на каждом шаге останавливаться можно сколько угодно. Каждый пункт безопасно повторить.

Шаги (порядок важен только между «образы» и «применить»):

1. **Рыба окружения** — **Настройка → Окружение и образы → Создать рыбу окружения (env-new)**. Спрашивает имя `ENV` (валидация: латиница/цифры/`-`/`_`), создаёт `environments/<ENV>.{mk,yaml}`, `k8s/config/<ENV>`, `values-<ENV>.yaml` в каждом сервисе. После успеха предлагает переключить сессию на новый ENV. CLI-эквивалент: `make env-new ENV=…`. Цель идемпотентна.
2. **Доступ к кластеру:**
   - **Удалённо:** `SSH_HOST` / `SSH_KEY` в `environments/<ENV>.mk` (правка вручную; шаблон создан на шаге 1), затем **Окружение и образы → MicroK8s: установить/проверить на ноде**, далее **Окружение и образы → Kubeconfig: получить с удалённого сервера**. CLI: `make microk8s-setup ENV=…`, `make kubeconfig-fetch ENV=…`.
   - **Локально:** **Окружение и образы → Kubeconfig: локальный MicroK8s** (без SSH). CLI: `make kubeconfig-microk8s-local ENV=…`.
3. **Образы:** **Окружение и образы → Образы: pull + сохранить tar**, затем **Образы: загрузить в локальный registry** (local) или **загрузить на удалённый сервер** (remote). На вопрос «Ограничить одним сервисом?» по умолчанию — нет (нужны все). CLI: `make images-save ENV=…` → `make images-push ENV=…` или `make images-push-remote ENV=…`.
4. **Состав сервисов** (опционально, если на этом ENV не весь набор) — **Сценарии → Конфигурирование сервисов**. Зафиксирует политику в `<ENV>.mk` и покажет diff. Подробности — Сценарий 6.
5. **Развёртывание** — **Каждый день → Применить изменения** (helm up + apps-apply). См. [общие замечания](#infra-tui-общие-замечания) про `ENABLED_SERVICES` / `SKIP_APPS_APPLY`. CLI: `make up ENV=…`.
6. **Sanity** — **Каждый день → Состояние и логи → Полная диагностика стека**. CLI: `make doctor ENV=…`.

Восстановление существующего ENV из git + бэкапов — [disaster-recovery.md](./disaster-recovery.md).

## Сценарий 2: новое приложение с подмножеством сервисов

**Цель:** приложению нужны, например, только Postgres и Redis — без Kafka/ClickHouse.

Самый простой путь — единый wizard: **Сценарии → Подключить приложение** (7 шагов: имя → шаблон → редактирование → конфигуратор → merge → apps-apply → опц. clone/hostpath).

Шаги:

1. Платформенные сервисы уже подняты (см. сценарий 1).
2. Шаблон каталога `apps/conf/<app>/` + опционально запись в `apps/registry.yaml` — `make apps-conf-template APP=<app>`. TUI: шаг 2 wizard'а.
3. Заполнить секреты и поля (`enabled`, `app_ns`, `redis_db`, `repo`) — конфигуратор `runConfigure` (запускается из шага 4 wizard'а, либо отдельно `node scripts/configure-infra.mjs`). Просмотр merge: `make apps-merge-print` — шаг 5 wizard'а.
4. Применить в кластер — `make apps-apply ENV=…`. TUI: шаг 6 wizard'а либо **Каждый день → Учётки и топики → Применить учётки**.
5. Сервис-специфичные операции — цели `*-app-create` / `*-app-show-creds` / `pg-app-psql` и пр. TUI: **Каждый день → Учётки и топики → движок**. APP при входе спрашивается один раз и кэшируется per-ENV.

Для `ENV=local` с hostPath исходников: `make apps-local-src-helm-sets ENV=local APP=…` (строки `--set` для чарта приложения; шаблон тома задан в самом чарте). Маунт apps/src в pod приложения — шаг 7 wizard'а (только при `ENV=local`).

## Сценарий 3: смена учёток, ротация секретов, правка квот

**Цель:** новый пароль приложения, другой Redis DB, другие параметры топика Kafka.

1. Новые секреты в `apps/conf/<app>/` через конфигуратор (`node scripts/configure-infra.mjs`); применение — `make apps-apply ENV=…` (TUI: **Каждый день → Учётки и топики → Применить учётки**).
2. Просмотр текущих кредов: `*-app-show-creds`. TUI: **Каждый день → Учётки и топики → движок → Показать креды и эндпоинт**. APP берётся из контекста сессии (спрашивается один раз).
3. Kafka-топики: `kafka-topic-alter` / `-describe` / `-list`. TUI: **Каждый день → Учётки и топики → Топики Kafka**. По умолчанию префикс листинга — `<APP>.`.

При ротации секретов и правках `apps/registry.yaml` — обновить env-архив (см. сценарий 5).

## Сценарий 4: здоровье сервисов и инциденты

**Цель:** понять статус релиза и упирается ли расчёт в ресурсы ноды.

Единая точка для всего сценария — **Каждый день → Состояние и логи** (8 пунктов в одном подменю):

1. Комплексная сводка: `make doctor ENV=…` (тулинг + кластер + helm vs helmfile + rollouts + per-app verify). TUI: **Полная диагностика стека**.
2. Per-сервис: `<svc>-status` / `-logs` / `-shell`. TUI: **Сводка по кластеру** / **Логи сервиса** / **Shell в поде сервиса** (выбор компонента после).
3. Нагрузка ноды и события: `top-totals`, `monitoring-pod-events`, `monitoring-describe-pod`. TUI: **Загрузка узлов** / **События namespace monitoring** / **События конкретного пода** / **Describe конкретного пода**.
4. Постоянный слой наблюдаемости — [Netdata](../../README.md#мониторинг-netdata) в `ENABLED_SERVICES` (включается мультивыбором при **Каждый день → Применить изменения**).

## Сценарий 5: регулярные бэкапы и проверка восстановления

**Цель:** свежие бэкапы стейтфул-сервисов и периодический smoke-тест применимости.

Единая точка для всего сценария — пункт **Бэкапы** в главном меню (5 операций в одном подменю).

1. По графику (cron / systemd timer / CI) для каждого `ENV`:
   - **Полный** (всё разом): `make backup-all ENV=…` + `make env-backup ENV=…`. TUI: **Бэкапы → Сделать бэкап: полный** (с предложением сразу запустить env-backup).
   - **Все стейтфул-сервисы только:** `make backup-all ENV=…` — последовательно `postgres-backup`, `redis-backup`, `kafka-backup-meta`, `minio-backup-meta`, `clickhouse-backup`, `rabbitmq-backup-defs` (учитывает `ENABLED_SERVICES`/`EXCLUDE_SERVICES`). Артефакты — в `<service>/backups/` (gitignored). Подробности — `<service>/BACKUP.md`. TUI: **Бэкапы → Сделать бэкап: выборочно** (или «полный» без env-шага).
   - **Платформенные Secrets и `apps/conf/`:** `make env-backup ENV=…` — `environments/backups/<env>-YYYYMMDD-HHMMSS.tar.gz`. Без него восстановление с нуля невозможно. TUI: **Бэкапы → Сделать бэкап: только секреты**.
2. Хранить артефакты **вне той же ноды** (rsync/rclone в S3, отдельный сервер, при необходимости шифрование — см. [secrets-management.md](./secrets-management.md), [onboarding-admin.md](../onboarding-admin.md)).
3. Ротация — оператор задаёт RPO/RTO, чистит старые архивы.
4. **Smoke-тест восстановления** в local/staging: `make <svc>-restore BACKUP_FILE=… ENV=…` (для каждого сервиса — соответствующий `<service>/BACKUP.md`). TUI: **Бэкапы → Восстановить сервис из архива** — выбор сервиса, ввод `BACKUP_FILE`, двойное подтверждение. Без smoke-теста «бэкап есть» ≠ «бэкап применим».
5. При смене паролей/ACL и правках `apps/registry.yaml`/`apps/conf/<app>/` — новый `env-backup`, иначе старый архив содержит устаревшие Secrets.

Расписание задаётся в системе оператора, не в TUI. Полный DR-runbook — [disaster-recovery.md](./disaster-recovery.md).

## Сценарий 6: конфигурирование набора сервисов

**Цель:** управлять составом релизов helm в одном или нескольких ENV — добавить новый сервис в работающий кластер, разгрузить local, держать разные стеки на local и prod, зафиксировать политику в `environments/<ENV>.mk`.

Состав на каждом ENV определяют **две переменные** в `environments/<ENV>.mk`:

- `ENABLED_SERVICES` — whitelist (если задана — только эти сервисы);
- `EXCLUDE_SERVICES` — blacklist (исключить из набора);
- если обе пустые — **весь набор** из `helmfile.yaml.gotmpl`.

`Makefile` инклюдит `<ENV>.mk` через `-include`, поэтому переменные подхватываются автоматически: `make up ENV=…` без флагов уважает то, что записано.

### Wizard

**Сценарии → Конфигурирование сервисов** делает только две вещи: фиксирует политику в `<ENV>.mk` и показывает diff. Apply остаётся за **Каждый день → Применить изменения**, точечный destroy — за **Окружение и образы → Helm: уничтожить релизы (опасно)**.

Шаги wizard'а:

1. Печатает текущее состояние `<ENV>.mk` (`ENABLED_SERVICES=…`, `EXCLUDE_SERVICES=…`, либо «—» если не заданы).
2. Спрашивает режим: **Весь набор** / **Только указанные** / **Кроме указанных** / **Отмена**.
3. Если не «весь набор» — мультивыбор сервисов (`HELM_OPTIONS`, включая Netdata); по умолчанию отмечено то, что уже было записано.
4. Записывает в `<ENV>.mk`:
   - «весь набор» → удаляет обе переменные (`clearHelmServiceVarsMk`);
   - «только указанные» → пишет `ENABLED_SERVICES`, удаляет `EXCLUDE_SERVICES`;
   - «кроме указанных» → пишет `EXCLUDE_SERVICES`, удаляет `ENABLED_SERVICES`.
5. Опционально показывает `make diff` (без env-флагов — читает из `.mk`).
6. Подсказывает следующие шаги.

### Расширить состав (добавить сервис)

1. **Сценарии → Конфигурирование сервисов** → добавить сервис в выбор → diff (увидите новый релиз, который будет создан).
2. **Окружение и образы → Образы: pull + сохранить tar** → **загрузить в локальный registry** или **на удалённый сервер**. На вопрос «Ограничить одним сервисом?» — **да**, выбрать новый `<svc>`.
3. Для Kafka — **Окружение и образы → Kafka: bootstrap-подключение к новому брокеру** (`make kafka-bootstrap`).
4. **Каждый день → Применить изменения**.
5. Учётки приложений для нового сервиса — **Каждый день → Учётки и топики → движок → Создать учётку**.

### Сузить состав (убрать сервис)

1. **Сценарии → Конфигурирование сервисов** → исключить сервис из выбора → diff (увидите релиз с пометкой «removed»).
2. Решить, что делать с уже установленным релизом:
   - оставить как есть (диф не применять) — политика в силе только для будущих `up`;
   - **уничтожить:** **Окружение и образы → Helm: уничтожить релизы (опасно)** → выбрать `<svc>` → двойной confirm.
3. **PVC и данные:** уничтожение helm-релиза **не удаляет PVC автоматически**, но при `storageClass: ""` фактический backing — `microk8s-hostpath` с `reclaimPolicy: Delete`, и удаление PVC уничтожит данные. Подробности — [storage-class.md](./storage-class.md) и `<service>/TROUBLESHOOTING.md`.

### Разные составы по ENV (local vs prod)

Wizard работает per-ENV: переключите сессию через **Сессия → Сменить ENV** и зафиксируйте свою политику для каждого окружения. Одна кодовая база, разные `values-<ENV>.yaml` и разный набор релизов. Подробнее — [README — окружения](../../README.md#окружения-localprodstaging).

### Полный снос окружения

`make down ENV=…` (TUI: **Окружение и образы → Helm: уничтожить релизы (опасно)** → ответ «весь набор» в мультивыборе → двойной confirm). Осторожно с PVC: см. предупреждения выше и в `<service>/README.md`.

## Сценарий 7: отключить приложение

**Цель:** вывести приложение из эксплуатации (платформенные сервисы остаются для других APP).

В реестре `enabled: false`, затем `make apps-apply`; при необходимости явное удаление учёток (`*-app-drop` с подтверждением). TUI: **Сценарии → Отключить приложение** (4 шага: APP → `enabled=false` → `apps-apply` → опц. мультивыбор движков и `*-app-drop`). MinIO дополнительно спрашивает про удаление бакетов (`MINIO_REMOVE_BUCKETS=1`).

Отделяйте «снять с helm» от «очистить учётки и данные приложения» — это разные шаги. Удаление платформенного сервиса целиком — Сценарий 6 (раздел «сузить состав» / «полный снос»).
