# Приёмка infra-control

## Definition of Done

1. **Покрытие целей.** Каждая цель из корневого `.PHONY` в `Makefile`, кроме `help`, `infra-lab` и **`infra-control-parity-check`** (мета-таргет для CI/ручной сверки, не операция инфры), отражена в [targets.json](targets.json); также явно включены операции Postgres без отдельного `.PHONY` в первой строке: `postgres-backup`, `postgres-restore`, `postgres-delete-pvcs`, `postgres-recreate-prep`, `pg-app-verify`.
2. **Запуск целей.** Из TUI любой лист в итоге вызывает в корне репозитория сценарий с выбранным именем цели и собранными переменными окружения (или без лишних), см. [`scripts/lib/make.mjs`](../../scripts/lib/make.mjs).
3. **Clack-преемственность.** Выбор действий только через `@clack/prompts` (`select`, `confirm`, `text`, …); сообщения результатов через `log.*`; **`Ctrl+C`/отмена** не роняют процесс трассировкой без `try/catch` на уровне циклов меню.
4. **Опасные цели.** Для `dangerous: true` в `targets.json` — явное второе подтверждение (или ввод ключевого слова, например `DROP`, перед удалением учётки) плюс, где это уже заложено в сценариях, внутренние дополнительные запросы.
5. **Потоковые процессы.** Для целей с `sigint: true` передаётся `handleSigint: true` в `runMake` (follow логов, port-forward).

## Автоматическая проверка паритета

```bash
node docs/infra-control/parity-check.mjs
```

Ожидается код выхода `0`; при расхождениях — список отсутствующих или лишних целей относительно `targets.json`.

## Ручные smoke-сценарии (без кластера допустим частичный fail)

После любой доработки ветки меню имеет смысл пройти хотя бы:

| Сценарий | Ожидание |
|----------|----------|
| Запуск `node scripts/infra-lab.mjs`, корень, **Выход** | Корректный `outro`, код 0 |
| **Конфигурирование** → **Приложение** → **Конфигуратор** → отмена | Возврат в меню без падения процесса |
| **Управление** → **Среда** → **Кластер** → status с неверным KUBECONFIG | Понятная ошибка `kubectl`; exit ≠ 0, меню живо |
| **Управление** → **Сервис** → **Helm** → diff → выбор сервисов | В лог перед запуском выводится сводка с именем цели (например `diff` и переменные) |

Полная функциональная проверка (helm apply, образы на registry, microk8s-setup) выполняется вручную в реальной инфраструктуре при необходимости.
